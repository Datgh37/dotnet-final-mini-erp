CHƯƠNG 1: TỔNG QUAN & KIẾN TRÚC
1. Phần 1: Tổng Quan & Kiến Trúc
1.1. Giới Thiệu Đề Tài
Ứng dụng: Mini ERP là một hệ thống quản lý tài nguyên doanh nghiệp (ERP) phiên bản tinh gọn cho cửa hàng nhỏ.
Hệ thống bao gồm: MiniERP_API (Web API backend xử lý logic nghiệp vụ); MiniERP_WebView (Frontend Web MVC cho giao diện người dùng)
Công nghệ cốt lõi:
• Backend (MiniERP_API)
o Runtime: .NET 10.0
o Framework: ASP.NET Core Web API
o Database: SQL Server
o Data Access: ADO.NET (Microsoft.Data.SqlClient)
o Mapping: AutoMapper (DTO mapping)
o API Documentation: NSwag (Swagger UI)
o Architecture Pattern: Repository & Service Pattern
• Frontend (MiniERP_WebView)
o Framework: ASP.NET Core MVC
o Runtime: .NET 10.0
o Styling: Bootstrap 5 + HTML5
Link Source Code: https://github.com/Datgh37/dotnet-final-mini-erp
Các nghiệp vụ chính:
STTNghiệp vụMô tả1Quản lý Người dùng & Phân quyềnQuản lý tài khoản người dùng, băm mật khẩu và phân quyền vai trò (Roles) cho nhân viên.2Quản lý sản phẩmCRUD thông tin sản phẩm, danh mục hàng hóa, thương hiệu và đơn vị tính.3Quản lý đối tácCRUD thông tin chi tiết khách hàng và nhà cung cấp.4Nhập – Xuất khoThực hiện phiếu nhập kho (khi mua), phiếu xuất kho (khi bán), lưu nhật ký biến động và theo dõi số lượng tồn kho thực tế.5Quản lý mua hàngTạo đơn đặt hàng từ nhà cung cấp (Purchase Order), theo dõi ngày dự kiến và ngày nhận hàng thực tế.6Quản lý bán hàngTạo đơn hàng bán (Sales Order), chi tiết đơn hàng, quản lý phương thức thanh toán và địa chỉ giao hàng.7Tính toán doanh thuTự động tính tổng tiền đơn hàng dựa trên số lượng, đơn giá và chiết khấu.8Thống kê báo cáoTổng hợp và báo cáo doanh thu theo các khoảng thời gian (ngày, tháng, năm).9Bảo toàn dữ liệuXóa mềm (Soft Delete) tất cả dữ liệu (không xóa vật lý)1.2. Kiến Trúc Hệ Thống (3-Layer Architecture)

Giải thích:
• Presentation Layer: Nhận yêu cầu từ người dùng thông qua API, kiểm tra dữ liệu đầu vào và trả về JSON response.
• Business Logic Layer: Chứa các quy trình nghiệp vụ cốt lõi, thực hiện tính toán và điều phối các hành động giữa tầng API và tầng dữ liệu.
• Data Access Layer: Thực hiện kết nối và tương tác trực tiếp với cơ sở dữ liệu bằng ADO.NET thông qua các mẫu Repository.
• Database Layer: Lưu trữ dữ liệu vật lý và xử lý các nghiệp vụ phức tạp trực tiếp bên trong SQL Server bằng Stored Procedures.

1.4. Luồng Xử Lý Request
Client (Postman/Frontend)
    │
    ▼
[HTTP Request] → Controller (Validate + Map DTO)
                      │
                      ▼
                 Service (Business Logic)
                      │
                      ▼
                 Repository (ADO.NET → SQL Server)
                      │
                      ▼
                 [SQL Query Execute]
                      │
                      ▼
                 Repository (Map DataReader → Model)
                      │
                      ▼
                 Service (Process + Return)
                      │
                      ▼
                 Controller (Map → Response DTO)
                      │
                      ▼
              [HTTP Response JSON]

2.1. Nguyên Tắc Soft Delete
Tất cả các bảng đều có cột IsDeleted (BIT, default 0).
Khi "xóa", chỉ UPDATE IsDeleted = 1, KHÔNG dùng DELETE.
Tất cả câu SELECT đều thêm WHERE IsDeleted = 0.

2.4. Stored Procedures Quan Trọng
2.4.1. Tạo đơn bán hàng & Trừ kho (sp_CreateSalesOrder)
• Chức năng: Dùng để xử lý trọn gói một đơn hàng bán ra.
• Các bước thực hiện:
o Tạo bản ghi mới trong bảng SalesOrders.
o Đọc danh sách sản phẩm từ định dạng XML và chèn vào bảng SalesOrderItems.
o Kiểm tra tồn kho từng sản phẩm; nếu thiếu sẽ báo lỗi và hủy toàn bộ giao dịch.
o Tự động trừ số lượng trong bảng Products.
o Ghi nhật ký xuất kho vào bảng StockMovements.
o Tự động tính toán lại tổng tiền của đơn hàng.
2.4.2. Xác nhận nhập hàng (sp_ReceivePurchaseOrder)
• Chức năng: Dùng khi hàng từ nhà cung cấp về tới kho.
• Các bước thực hiện:
o Cập nhật trạng thái đơn mua hàng (PurchaseOrders) thành 'RECEIVED'.
o Cập nhật ngày nhận hàng và người nhận.
o Tự động cộng thêm số lượng tồn kho vào bảng Products.
o Ghi nhật ký nhập kho vào bảng StockMovements.
2.4.3. Báo cáo doanh thu (sp_GetRevenueReport)
• Chức năng: Kết xuất dữ liệu thống kê kinh doanh trong một khoảng thời gian.
• Các bước thực hiện:
o Lọc các đơn hàng có trạng thái 'COMPLETED' và không bị xóa.
o Nhóm dữ liệu theo ngày.
o Tính tổng số lượng đơn hàng và tổng doanh thu của từng ngày trong khoảng từ ngày FromDate đến ToDate.
2.4.4. Xóa mềm tổng quá (sp_SoftDelete)
• Chức năng: Một công cụ dùng chung để xóa dữ liệu an toàn cho tất cả các bảng.
• Các bước thực hiện:
o Nhận tên bảng và ID cần xóa làm tham số.
o Thay vì xóa vĩnh viễn, SP này cập nhật cột IsDeleted = 1 và cập nhật thời gian UpdatedAt.
o Giúp bảo toàn dữ liệu để phục vụ việc đối soát và báo cáo lịch sử.

CHƯƠNG 4: KHỞI TẠO DỰ ÁN & CẤU HÌNH
Step 1: Tạo Solution và Projects
• Dự án được tổ chức theo mô hình nhiều project trong cùng một Solution.
• Solution chính bao gồm hai project: MiniERP_API và MiniERP_WebView.
• MiniERP_API sử dụng ASP.NET Core Web API để xử lý backend và kết nối database.
• MiniERP_WebView sử dụng ASP.NET Core MVC để hiển thị giao diện người dùng.
• Các bước tạo Solution gồm: tạo Blank Solution, thêm Web API project và MVC project.
Step 2: Cấu hình Connection String
• Connection String được cấu hình trong file appsettings.json.
• Thông tin bao gồm Server, Database, Authentication và TrustServerCertificate.
• Database script gồm Schema.sql và SeedData.sql để tạo bảng và dữ liệu mẫu.
Ví dụ cấu hình appsettings.json trong project:
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.;Database=MiniERP;Trusted_Connection=True;TrustServerCertificate=True"
  }
}
Giải thích:
• Server=. : sử dụng SQL Server local
• Database=MiniERP : tên database
• Trusted_Connection=True : dùng Windows Authentication
• TrustServerCertificate=True : bỏ kiểm tra SSL certificate
Ví dụ đọc Connection String trong project:
public SqlConnectionFactory(IConfiguration configuration)
{
    _connectionString = configuration.GetConnectionString("DefaultConnection");
}
Step 3: Tạo SqlConnectionFactory (DAL)
• SqlConnectionFactory giúp quản lý kết nối SQL tập trung.
• Factory sử dụng IConfiguration để đọc Connection String.
• Repository sẽ gọi CreateConnection() để tạo SqlConnection mới.
• Mô hình DAL giúp giảm lặp code và dễ bảo trì.
Ví dụ class SqlConnectionFactory:
using Microsoft.Data.SqlClient;
using System.Data;

public class SqlConnectionFactory
{
    private readonly string _connectionString;

    public SqlConnectionFactory(IConfiguration configuration)
    {
        _connectionString = configuration.GetConnectionString("DefaultConnection");
    }

    public IDbConnection CreateConnection()
    {
        return new SqlConnection(_connectionString);
    }
}
Step 4: Đăng ký Dependency Injection (Program.cs)
• Program.cs là nơi cấu hình service và middleware.
• Dependency Injection giúp tự động inject object giữa Controller, Service và Repository.
• SqlConnectionFactory thường được đăng ký bằng AddSingleton.
• Repository và Service thường dùng AddScoped.
Ví dụ code Program.cs trong project:
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

builder.Services.AddSingleton<SqlConnectionFactory>();

builder.Services.AddScoped<IProductRepository, ProductRepository>();
builder.Services.AddScoped<IProductService, ProductService>();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();
app.UseAuthorization();

app.MapControllers();

app.Run();
Ý nghĩa:
• AddSingleton: tạo duy nhất 1 instance trong toàn bộ application
• AddScoped: tạo mới theo mỗi HTTP Request
• MapControllers(): ánh xạ API Controller
• UseSwagger(): bật Swagger để test API
Step 5: Tạo BaseEntity (Models)

Ví dụ BaseEntity:
public class BaseEntity

5.1 Models
      Models là các class đại diện cho dữ liệu của hệ thống và ánh xạ với các bảng trong SQL Server.
      Các model được sử dụng trong hệ thống bao gồm:
• Product
• Category
• Order
• OrderDetail
• User
• Inventory
      Các model trong project thường được tổ chức trong thư mục Models và được sử dụng xuyên suốt toàn bộ hệ thống. Khi Repository truy vấn dữ liệu từ SQL Server, dữ liệu sẽ được chuyển đổi thành các object model tương ứng để Service và Controller có thể tiếp tục xử lý. Nhờ đó kiến trúc hệ thống trở nên rõ ràng và có tính đồng bộ cao hơn.
5.2 Vai trò của Models
• Đại diện dữ liệu database
• Trao đổi dữ liệu giữa các tầng
• Hỗ trợ mapping dữ liệu
• Chuẩn hóa cấu trúc hệ thống
5.3 BaseEntity
      Các model kế thừa BaseEntity để tái sử dụng các thuộc tính chung.
public class BaseEntity
{
    public int Id { get; set; }
    public DateTime CreatedAt { get; set; }
    public DateTime UpdatedAt { get; set; }
    public bool IsDeleted { get; set; }
}
      Khi một model như Product kế thừa BaseEntity thì model đó sẽ tự động có đầy đủ các thuộc tính dùng chung mà không cần khai báo lại. Cách tổ chức này giúp hệ thống dễ mở rộng hơn trong tương lai và hỗ trợ tốt cho việc quản lý dữ liệu cũng như theo dõi lịch sử thay đổi.
5.4 Data Access Layer (DAL)
      DAL là tầng chịu trách nhiệm truy cập dữ liệu và thao tác với SQL Server.
• Kết nối database
• Thực hiện CRUD
• Chạy SQL Query
• Tách biệt dữ liệu khỏi business logic
      Dự án sử dụng Repository Pattern để tổ chức Data Access Layer. Mỗi repository sẽ chịu trách nhiệm thao tác với một nhóm dữ liệu cụ thể. Ví dụ: ProductRepository quản lý dữ liệu sản phẩm, OrderRepository quản lý dữ liệu đơn hàng. Cách tổ chức này giúp hệ thống có cấu trúc rõ ràng và dễ dàng mở rộng thêm chức năng mới.
5.5 SqlConnectionFactory
      Project sử dụng SqlConnectionFactory để quản lý kết nối SQL tập trung.
public class SqlConnectionFactory
{
    private readonly string _connectionString;

    public SqlConnectionFactory(IConfiguration configuration)
    {
        _connectionString = configuration.GetConnectionString("DefaultConnection");
    }

    public IDbConnection CreateConnection()
    {
        return new SqlConnection(_connectionString);
    }
}
      Class này có nhiệm vụ đọc Connection String từ file appsettings.json và tạo đối tượng SqlConnection cho Repository sử dụng. Việc gom phần tạo kết nối vào một class riêng giúp tránh lặp code và tăng tính nhất quán trong toàn hệ thống.
5.6 Repository Pattern
      Repository Pattern giúp tách riêng tầng truy cập dữ liệu khỏi tầng nghiệp vụ.
• Controller gọi Service
• Service gọi Repository
• Repository truy cập SQL Server
• Repository trả dữ liệu về Service
public class ProductRepository : IProductRepository
{
    private readonly SqlConnectionFactory _factory;

    public ProductRepository(SqlConnectionFactory factory)
    {
        _factory = factory;
    }

    public async Task<IEnumerable<Product>> GetAll()
    {
        using var connection = _factory.CreateConnection();
        var sql = "SELECT * FROM Products";
        return await connection.QueryAsync<Product>(sql);
    }
}
      Trong đoạn code trên, ProductRepository sử dụng SqlConnectionFactory để tạo kết nối database. Sau đó Repository thực hiện câu lệnh SQL để lấy toàn bộ dữ liệu sản phẩm từ bảng Products. Kết quả truy vấn được ánh xạ thành danh sách Product và trả về cho Service Layer.
      
CHƯƠNG 6: SERVICE LAYER
6.1 Service Layer
      Business Logic Layer hay còn gọi là tầng xử lý nghiệp vụ là thành phần trung tâm của hệ thống. BLL nằm giữa Controller và Data Access Layer, có nhiệm vụ xử lý các quy tắc nghiệp vụ trước khi truy cập hoặc trả dữ liệu về cho client. Đây là nơi chứa các logic quan trọng của hệ thống ERP như tính toán dữ liệu, kiểm tra điều kiện, xử lý workflow và phối hợp nhiều repository khác nhau.
6.2 Vai trò của Service Layer
      Service Layer giúp giảm tải cho Controller bằng cách chuyển toàn bộ logic xử lý sang tầng riêng.
public class ProductService : IProductService
{
    private readonly IProductRepository _repository;

    public ProductService(IProductRepository repository)
    {
        _repository = repository;
    }

    public async Task<IEnumerable<Product>> GetAllProducts()
    {
        return await _repository.GetAll();
    }
}
      Trong ví dụ trên, ProductService nhận ProductRepository thông qua Dependency Injection. Khi Controller yêu cầu lấy danh sách sản phẩm, Service sẽ gọi Repository để truy vấn dữ liệu và trả kết quả về cho Controller.
6.3 Flow hoạt động
• Client gửi request
• Controller nhận request
• Controller gọi Service
• Service xử lý logic
• Repository truy cập database
6.4 Service Layer trong Project
      Trong project, Service Layer đại diện cho Business Logic Layer.

public class ProductService : IProductService
{
    private readonly IProductRepository _repository;

    public ProductService(IProductRepository repository)
    {
        _repository = repository;
    }

    public async Task<IEnumerable<Product>> GetAllProducts()
    {

6.5 Lợi ích của Business Layer
• Giúp Controller sạch hơn
• Có thể tái sử dụng logic
• Giúp hệ thống dễ maintain
• Có thể dễ dàng mở rộng hệ thống
• Unit test cũng sẽ dễ hơn
CHƯƠNG 7: API CONTROLLERS & DTOs
7.1 API Controller
      Controller là nơi nhận HTTP Request và trả HTTP Response.
• Nhận request từ frontend
• Gọi service xử lý
• Trả dữ liệu JSON
• Điều hướng endpoint

      Mỗi Controller thường đại diện cho một nhóm chức năng cụ thể của hệ thống. Ví dụ ProductsController quản lý API sản phẩm, OrdersController quản lý API đơn hàng. Controller sẽ gọi Service Layer để xử lý nghiệp vụ thay vì thao tác trực tiếp với database.

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private readonly IProductService _service;

    public ProductsController(IProductService service)
    {
        _service = service;
    }

    [HttpGet]
    public async Task<IActionResult> GetAll()
    {
        var products = await _service.GetAllProducts();
        return Ok(products);
    }
}
7.2 DTO và vai trò trong API
      DTO (Data Transfer Object) là object dùng để truyền dữ liệu giữa API và client.
• Ẩn dữ liệu nhạy cảm
• Giảm dữ liệu dư thừa
• Tăng bảo mật
• Tối ưu response

public class ProductDTO
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
}

      Việc sử dụng DTO giúp tăng tính bảo mật cho hệ thống vì có thể ẩn các trường dữ liệu nhạy cảm. Ngoài ra DTO còn giúp giảm dung lượng dữ liệu trả về và tối ưu hiệu suất API.
7.3 Swagger API
      Dự án Mini ERP sử dụng Swagger để hỗ trợ kiểm thử API trực tiếp trên trình duyệt. Swagger giúp lập trình viên dễ dàng xem danh sách endpoint, gửi request và kiểm tra response mà không cần sử dụng phần mềm bên ngoài.

builder.Services.AddSwaggerGen();

8.3. Tóm Tắt Kỹ Thuật Đã Sử Dụng
Kỹ thuậtNơi áp dụng**ASP.NET Core Web API**Presentation Layer (Controllers)**ADO.NET** (SqlConnection, SqlCommand, SqlDataReader)Data Access Layer**Dependency Injection**Program.cs – đăng ký Service/Repository**Repository Pattern**DAL – tách biệt logic truy vấn DB**3-Layer Architecture**API → BLL → DAL**Soft Delete**`IsDeleted = 1` thay vì DELETE**SQL Transaction**Inventory – đảm bảo toàn vẹn dữ liệu**DTO Pattern**Request/Response riêng biệt**Data Annotations**Validate input tại Controller8.4. Thứ Tự Triển Khai (Roadmap)
Tuần 1: Database + Models + DAL (Products, Categories)
    ↓
Tuần 2: BLL + API Controllers (Products, Customers) 
    ↓
Tuần 3: Orders + OrderDetails (nghiệp vụ phức tạp)
    ↓
Tuần 4: Inventory (Nhập/Xuất kho + Transaction)
    ↓
Tuần 5: Reports (Thống kê doanh thu) + Test + Hoàn thiện
